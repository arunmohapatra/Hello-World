In Terraform, the flags -refresh and -refresh-only control how Terraform syncs the state file with the real infrastructure before applying changes. They are often confusing because they look similar but behave differently.

I'll explain them clearly with examples.


1. -refresh=true vs -refresh=false

This flag controls whether Terraform checks the real infrastructure before planning/applying.

Default behavior

By default Terraform behaves like:

terraform plan -refresh=true
terraform apply -refresh=true

That means:

👉 Terraform first checks the actual cloud resources (AWS, Azure, etc.)
👉 Updates the Terraform state file if something changed outside Terraform.


Example

Suppose you created an EC2 instance using Terraform.

resource "aws_instance" "web" {
  instance_type = "t2.micro"
}
Case 1 — Someone changes infrastructure manually

Someone changes the instance type in AWS console:

t2.micro → t2.medium


Case A — -refresh=true
terraform plan -refresh=true

Terraform will:

Check AWS

See instance changed to t2.medium

Update state

Show plan to revert it back

Example plan:

~ instance_type: "t2.medium" -> "t2.micro"

So Terraform detects drift.

Case B — -refresh=false
terraform plan -refresh=false

Terraform will:

❌ NOT check AWS

It will trust the existing state file.

Result:

No changes. Infrastructure up-to-date.

But actually the infrastructure is different.

So drift is ignored.

But Terraform won't change the resource.

==================
3. What happens if you run normal terraform apply after refresh-only?

Now Terraform sees:

config: t2.micro
state:  t2.medium

So it will plan:

~ instance_type: "t2.medium" -> "t2.micro"

4. Summary Table
Flag	What it does	Infrastructure change
-refresh=true	Sync state with real infrastructure before planning	Yes
-refresh=false	Skip checking real infrastructure	Yes
-refresh-only	Update state to match infrastructure	❌ No changes

5. Practical DevOps Usage
-refresh=true

Normal pipeline execution.

terraform plan
terraform apply

-refresh=false

Used when:

• Infrastructure very large
• Refresh takes long time
• State already trusted

Example in CI/CD.

-refresh-only

Used for:

✔ Fixing state drift
✔ Syncing Terraform with manually changed resources
✔ Investigation/debugging

Example===================================================

Scenario

You created an AWS EC2 instance using Terraform.

resource "aws_instance" "web" {
  ami           = "ami-123456"
  instance_type = "t2.micro"
}

Run:

terraform apply

Result:

Component	    Status
AWS	           EC2 instance created
Terraform State	   Instance recorded
Terraform Code	   Instance defined

Everything is consistent.

==============================================
Step 1 — Someone deletes the instance manually

Someone goes to the Amazon EC2 console and deletes the instance.

Now the situation becomes:

Component	Status
AWS	❌ Instance deleted
Terraform State	✅ Still thinks instance exists
Terraform Code	✅ Instance defined

This is called infrastructure drift.

===========================================
Step 2 — Run normal Terraform apply
terraform apply

Terraform will refresh state first.

It detects:

aws_instance.web: resource missing

Plan will show:

+ create aws_instance.web

Terraform recreates the EC2 instance.

==========================================

Step 3 — Now run terraform apply -refresh=false

Command:

terraform apply -refresh=false

Terraform will:

❌ Skip checking AWS

It will trust the existing state file.

State says:

aws_instance.web exists

Terraform believes everything is correct.

So the output will be:

No changes. Infrastructure is up-to-date.

But in reality:

Component	       Status
AWS	           ❌ Instance does NOT exist
Terraform State	   ✅ Says it exists

So Terraform does nothing.

=====================================

Your infrastructure becomes inconsistent.

Terraform thinks:

Resource exists

But actually:

Resource is deleted

Next Run (Important)

If later someone runs:

terraform plan

(with refresh enabled)

Terraform will detect:

aws_instance.web: resource missing

Plan:

+ create aws_instance.web

Then the instance will be recreated.

=============================

      ↓

Step 2

      ↓

Step 3
terraform apply -refresh=false
      ↓

===================


When -refresh=false is used in real life

DevOps teams use it when:

• Infrastructure is very large
• Refresh takes too long (minutes/hours)
• They trust state consistency

Example in CI/CD pipelines.

But misuse can cause drift problems.

=====================refresh-only=======================
In Terraform, --refresh-only is not a true/false flag like -refresh.

It is a mode used with terraform plan or terraform apply.

So technically:

--refresh-only=true  ❌ not valid
--refresh-only=false ❌ not valid

You either:

use --refresh-only
or
do not use it

===========
1. What --refresh-only actually does

--refresh-only means:

👉 Update the Terraform state to match real infrastructure
👉 Do NOT apply any configuration changes

It only synchronizes the state file.

Example:

terraform apply --refresh-only

Terraform will:

Check real infrastructure

Update the state file

No resources will be created, deleted, or modified

====
Example

Terraform code:

resource "aws_instance" "web" {
  instance_type = "t2.micro"
}

Suppose someone manually changes instance type in Amazon EC2 console:

t2.micro → t2.medium

===
Case 1 — Normal terraform apply
terraform apply

Terraform detects drift:

~ instance_type: t2.medium → t2.micro

Terraform will change the instance back.

=====

Case 2 — terraform apply --refresh-only
terraform apply --refresh-only

Terraform will:

Detect instance is t2.medium

Update state file

Do nothing to infrastructure

Result:

Item	Value
AWS	t2.medium
Terraform state	t2.medium
Terraform code	t2.micro

State is updated but infrastructure not changed.


====
2. If you do NOT use --refresh-only

Example:

terraform apply

Terraform will:

1️⃣ Refresh state
2️⃣ Compare with configuration
3️⃣ Apply changes if needed


====

3. Summary
Command	What happens
terraform apply	Refresh + apply configuration changes
terraform apply --refresh-only	Only refresh state, no infrastructure change


====
4. When DevOps engineers use --refresh-only

Common situations:

✅ Fix state drift
✅ Sync state after manual changes
✅ Investigate infrastructure differences
✅ Safe audit without modifying resources

====
Simple way to remember
-refresh     → should Terraform check infrastructure?
--refresh-only → update state only
